# 日志、监测与事件流

## 1. 总体原则

Procora 的可观测性包含三条彼此隔离的数据通道：

- 控制事件：状态转换、命令结果和配置修订，要求可靠、有序。
- 日志帧：任务 stdout/stderr，要求持续排空、可查询和可限流。
- 指标快照：CPU、内存和 I/O 等高频数据，允许合并或降采样。

三者使用独立的队列、序号与保留策略。客户端日志渲染缓慢不能阻塞磁盘落盘，指标采样拥塞不能推迟进程退出处理。

## 2. 日志数据模型

每条 `LogFrame` 至少包含：

- `project_id`、`task_id`、`generation` 和 `run_id`。
- `stream`：stdout、stderr 或 Procora 生成的 system 事件。
- `seq`：该 run 内单调递增的帧序号。
- `observed_at`：daemon 观察时间，不能假装是应用业务时间。
- `bytes`：原始字节或经过明确解码标记的文本。
- `flags`：部分行、解码替换、截断等状态。

进程输出本质是字节流，不保证 UTF-8，也不保证一次读取对应一行。采集器先保存帧边界，再为 TUI 提供增量行解码；二进制或超长行必须有可预测的截断和展示策略。

## 3. 日志管线

```text
child stdout/stderr
  → 每流独立读取器
  → run 内序号分配
  → 内存尾部缓冲
  → 有界磁盘队列
  → 独立滚动文件写入器
  → 索引与保留执行器
  → 订阅分发器
```

关键约束：

1. stdout 和 stderr 必须持续排空，防止子进程因管道写满而阻塞。
2. 磁盘写入器落后时先触发告警并使用受限的缓冲策略，不能无限占用内存。
3. 单个客户端的订阅队列独立；溢出时发送 `Gap` 事件，客户端从日志存储按游标补读。
4. 如果磁盘不可写，任务默认继续运行并发出高优先级诊断；是否因日志故障停止任务应由显式严格策略控制。
5. 进程退出后需要短暂排空管道，再发布该 run 的日志完成事件。

## 4. Service 本地日志、自动滚动与保留

日志不由 Center 集中保存。每个 Service 在自身目录维护：

```text
<service>/.procora/logs/
  ├── service.log
  ├── service.log.<timestamp>-<sequence>.gz
  └── tasks/
      ├── <task>.log
      └── <task>.log.<timestamp>-<sequence>.gz
```

当前实现提供：

- 按大小滚动。
- 使用 gzip 自动压缩已关闭分片。
- 按每个活动日志的最大归档数量清理旧文件。
- 按全部归档总压缩字节数和最长保留时间清理旧文件。
- 在活动文件旁持久化“文件代次 + 字节偏移”索引。
- 旧游标跨越轮转或截断边界时显式返回 Gap，并从当前活动文件尾部有界恢复。
- Service 与各 Task 使用彼此独立的活动文件和归档集合。

默认策略为单活动文件 10 MiB、每个活动日志最多 5 个 gzip 归档、归档总计最多 50 MiB、最长保留 7 天。按固定时间主动滚动和归档全文查询仍是后续能力。

活动文件切换和索引更新必须采用原子元数据操作。清理器只删除已关闭且不再被读取器租用的分片。磁盘配额计算应包含索引和临时文件，并暴露当前占用量。

日志根目录固定在所属 Service 内，不使用 `PROCORA_HOME`。Task 文件名只能来自经过 `TaskId` 校验的标识，禁止使用未经净化的外部文本拼接路径。

## 5. 日志查询与 TUI

查询接口围绕稳定游标设计：

- `tail(task, run, lines)` 获取最近内容和起始游标。
- `read(cursor, limit, filter)` 分页读取历史。
- `subscribe(cursor)` 从指定位置开始实时订阅。
- 游标过期时返回最早可用位置，而不是静默缺失。

TUI 为选中 Task 保留本次查看会话读取到的完整视图缓存，按文件游标通过 Center IPC 以 64 KiB 分片按需续读，并在发生 Gap 时显示提示。CLI 日志命令同样逐片接收并立即输出，搜索与过滤只跨片保留尚未结束的半行，因此日志总量不会形成单个大发包或全量内存副本。常见 ANSI/SGR 控制序列会转换为 Ratatui 样式，不作为搜索文本显示。日志页默认跟随尾部，支持按实际视口翻页和跳到首尾；`/` 搜索、`n/N` 跳转匹配、`f` 过滤匹配行、连续两次 `C` 确认清空当前 Task 的活动日志和归档。macOS 展示对应的 Fn 组合键。鼠标捕获启用后，日志页纵向滚轮只移动日志而不改变 Task，触控板横向滚动与左右方向键一样横移文本，其他页面仍可用纵向滚轮切换 Task；上翻后会保持阅读位置，回到底部时恢复自动跟随。跨轮转归档的全文检索不在 UI 主循环中执行。

## 6. 资源监测模型

当前 `ResourceSnapshot` 聚合受管进程树的 CPU 使用率与常驻内存，并把不可用值显式留空。后续资源模型还应补充：

- CPU：采样区间内的归一化使用率和累计 CPU 时间。
- 内存：常驻内存，平台允许时提供虚拟内存。
- I/O：累计读写字节和可选吞吐率。
- 进程：受管进程树成员数、线程数或句柄数等可选项。
- 时间：单调时钟区间和墙钟展示时间。
- capability：每个指标是否原生、估算或不可用。

任务指标默认聚合整个受管进程树，而非只采样直接子进程。采样器需要处理进程在遍历期间退出、PID 复用和树成员变化；不完整快照应带标记，不伪造为精确值。

## 7. 采样与聚合

当前实现已经把 TUI 的 500ms 状态刷新、50ms 日志续读与 1s 资源采样分层。日志 IPC 单批最多读取 64 KiB，客户端只有需要更多内容时才携带游标续读；磁盘线程仍可把同一 Task 的连续输出合并后追加，降低快速小块输出的文件元数据与游标更新开销。`ServiceHost` 在一次系统进程刷新后批量聚合全部活动 Task 根，多个快照请求在慢周期内共享缓存；Task 启停会立即使缓存失效，没有活动根时不读取系统进程表。这避免任务数和观察客户端数直接放大全系统扫描次数。

后续继续按以下层次扩展：

- 活跃 TUI 可见任务按正常间隔采样（当前按 Service 统一 1s）。
- 后台无客户端时降低频率。
- 退出任务停止系统采样，只保留最终累计值。
- TUI 图表按窗口聚合 min/max/avg，避免每个原始点都跨 IPC 传输。

采样间隔应有最小值，防止配置错误造成高系统开销。一次采样超过间隔时禁止重叠启动下一轮，应记录延迟并跳过或合并周期。

首版监测只提供观测与告警，不执行 CPU/内存限制。资源限制属于平台差异较大的控制能力，需要 cgroup、Job Object 等独立设计后增加。

## 8. 事件流与一致性

控制事件由 daemon 分配全局 `event_seq`。新客户端连接流程：

1. 服务端记录当前序号并生成一致性状态快照。
2. 客户端接收快照和快照序号。
3. 服务端发送序号之后的增量事件。
4. 若缓冲窗口无法覆盖，则要求客户端重新获取快照。

日志使用 run 内游标，指标使用时间窗口游标，不强行共享控制序号。协议需要提供关联身份，使客户端能丢弃已被新 generation 替换的旧遥测。

## 9. 背压策略

| 通道 | 溢出策略 | 原因 |
| --- | --- | --- |
| 引擎控制 | 拒绝新外部命令或施加上游背压，不丢内部退出事件 | 关系到状态正确性 |
| 日志采集 | 有界缓冲、磁盘故障告警、订阅端可出现 Gap | 必须优先排空进程管道 |
| 指标采样 | 合并最新值或跳过周期 | 指标是可重建遥测 |
| 客户端状态事件 | 断开过慢会话并要求快照重连 | 不能阻塞全局事件循环 |

每个队列需要暴露长度、丢弃/合并次数和最后拥塞时间。默认容量必须通过压力测试确定，而不是只使用依赖库默认值。

## 10. 隐私与脱敏

- 默认不把完整环境变量写入诊断日志。
- 标记为 secret 的值在配置、错误链、协议和 TUI 中统一脱敏。
- 命令参数可能包含密钥，展示策略应允许按参数位置或模式隐藏。
- 日志正文由任务产生，Procora 无法保证自动识别所有密钥；文档应提示用户避免输出秘密。
- 导出诊断包时默认只包含结构化状态、能力信息和截断后的脱敏日志。
