# 实现路线图

## 1. 范围原则

路线图按可验证的纵向能力推进，不以“创建完所有 crate”作为完成标准。每个阶段都应交付可运行切片、测试和同步文档。中心服务器、高频 CLI 和首个真实 Task 运行闭环已经完成；下一主线是 P2 热更新与平台集成。

## 2. 阶段 0：架构与工作区基线

目标：把当前最小工程整理为可持续演进的单一 crate。

交付物：

- 建立第一批必要 crate：core、config、engine、process、platform、cli。
- 配置统一 lint、格式、测试和文档检查。
- 定义领域 ID、错误分类、时钟与配置修订等基础类型。
- 建立 `tests/` 结构和跨平台 CI。
- 为关键兼容性决策建立 ADR 目录。

验收条件：

- Linux、macOS、Windows 上单一 crate 构建和空测试通过。
- 模块依赖符合[总体架构](architecture.md)中的单向规则。
- 不存在提前占位但没有明确消费者的公共接口。

## 3. 阶段 1：配置与任务图

目标：在不启动进程的前提下完整编译任务配置。

交付物：

- YAML、TOML、JSON 统一反序列化与错误定位。
- `RawProject → ProjectSpec → TaskGraph` 管线。
- 默认值、profile、变量和路径解析的最小闭环。
- 环检测、依赖条件与语义差异计算。
- `procora validate` 和脱敏的 effective config 输出。

验收条件：

- 三种数据格式通过等价配置契约测试。
- 非法配置一次返回可操作的错误集合。
- 1000 节点的合法任务图可在合理时间与内存内完成编译。

## 4. 阶段 2：前台进程监管

目标：通过前台命令可靠运行带依赖的本地任务。

交付物：

- 单写者引擎、状态机和启动/反向停止调度。
- 三平台子进程启动、退出观察和进程树回收。
- `never`、`on-failure`、`always` 重启及有界退避。
- 基础健康检查和 Ctrl-C 正常关闭。
- 结构化控制事件输出。

验收条件：

- 三平台完成依赖启动、失败重启和整树停止端到端测试。
- 旧 run 的迟到事件不能改变新 run 状态。
- 前台关闭后不遗留可确认属于本次运行的子进程。

## 5. 阶段 3：日志与资源监测

目标：让任务运行过程可持续观察且有明确资源上限。

交付物：

- stdout/stderr 非阻塞采集、内存尾部和 Service 目录内滚动文件。（文件存储与 gzip 轮转已完成）
- 日志游标、历史读取、保留清理与磁盘故障降级。
- CPU、内存、I/O 的进程树采样及能力标志。
- 事件、日志和指标的独立有界通道及拥塞指标。

验收条件：

- 高速输出不会因慢观察者阻塞任务。
- 日志 gzip 滚动、数量/时间/总字节保留和持久文件游标已通过测试；轮转后的旧游标会显式报告 Gap。
- 三平台指标缺失被显式表达，采样器不因进程退出竞态崩溃。

## 6. 阶段 4：daemon、IPC 与 TUI

目标：支持后台运行、重连和完整观察体验。

交付物：

- 当前用户唯一 Center、多个 ServiceHost，以及保存当前状态和历史的 SQLite 注册表。（基础切片已完成）
- Unix domain socket / Windows Named Pipe 本地 IPC。（JSON 行请求已完成）
- 协议握手、状态快照、增量事件和日志续读。
- daemon 生命周期、当前用户端点权限和恢复元数据。
- TUI 任务列表、依赖图、详情、日志、指标和命令反馈页面。
- 具有增量观察能力的前端会话；控制动作由握手权限决定，不把“观察”固定解释成只读。

验收条件：

- TUI 退出不影响 daemon 和任务。
- 客户端断线重连后能恢复一致状态与日志位置。
- 过慢客户端不会拖慢引擎，并收到明确的重同步指令。
- 小终端、无颜色和非交互输出具备降级路径。

## 7. 阶段 5：热更新与任务定义源

目标：安全获取、拉取并应用任务定义变化。

交付物：

- 本地入口及 include 闭包监听。
- 候选修订、语义差异预览、确认和应用流程。
- 配置 generation 与任务按需重启。
- Python 配置辅助进程协议和可信代码提示。
- Git 定义源原型；HTTP 源是否进入该阶段由威胁模型评审决定。

验收条件：

- 写入中的临时无效配置不会破坏当前修订。
- 纯展示/采样变更不重启进程，身份字段变更只重启受影响闭包。
- Python 配置超时、崩溃或输出非法内容时保持旧修订。
- 远程候选默认不自动执行，来源和内容哈希可审计。

## 8. 阶段 6：稳定化与发布

目标：形成可直接下载、可诊断的首个预览版，不维护平台原生安装包。

交付物：

- 配置、协议和持久化模式版本策略。
- GitHub Actions 为 Linux、macOS、Windows 的 x86_64 与 ARM64 共六个目标构建二进制并发布到 Releases。
- macOS/Linux 一键安装 shell 脚本、Windows 一键安装 PowerShell 脚本、shell completion 和升级说明。
- 诊断包、崩溃恢复、长期运行与压力测试。
- 用户指南、配置参考、故障排查和安全说明。
- 性能基线与已知平台限制清单。

验收条件：

- 连续运行与高日志吞吐场景无持续资源泄漏。
- 六目标下载、脚本安装、首次运行和后台启停在对应平台经过验证。
- 所有未实现能力在 CLI、TUI 和文档中明确显示，不做静默降级。

## 9. 首个可用版本的取舍

首个可用版本应优先保证：任务图正确、进程树可回收、日志不阻塞、状态可恢复观察。以下能力应推迟，除非基础语义已稳定：

- 跨主机集群调度。
- 基于网络的远程控制。
- 把每个 Service/Task 分别安装为 systemd、launchd 或 Windows Service；Center 的当前用户级自启动已由 `enable/disable` 提供。
- 内核级资源配额和隔离。
- 插件内执行任意第三方代码。
- 复杂依赖布尔表达式和动态任务生成。

## 10. 当前下一步

`Center → ServiceHost → Task` 已形成运行闭环；安全 include 闭包、受控 Python 配置、固定提交 Git 候选源、配置监听去抖、SHA-256 候选、语义差异、显式确认、按影响 Task 对账与失败回退，以及供应链、MSRV、性质/日志压力门禁和兼容性 ADR 也已落地。下一步补 Center/CLI 的持久远端注册与凭据边界；工程质量主线继续补长期运行和受控 Git/SSH 故障注入。具体上游对照和取舍见 [`process-compose-study.md`](process-compose-study.md)。
