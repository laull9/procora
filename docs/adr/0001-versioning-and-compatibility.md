# ADR 0001：配置、IPC 与持久化版本兼容策略

- 状态：已接受
- 日期：2026-07-16
- 决策范围：公共配置模式、本地 IPC、SQLite 注册表

## 背景

Procora 的配置会启动本机代码，IPC 会执行生命周期控制，SQLite 会跨版本恢复服务运行期望。这三种边界的兼容错误都可能比普通解析失败更危险：旧字段被猜错会改变进程语义，旧客户端误发控制请求会破坏状态，降级程序误读新数据库会覆盖无法理解的数据。

项目当前分别使用配置顶层 `version: 1`、`PROTOCOL_VERSION = 7` 和 SQLite `PRAGMA user_version = 1`。配置候选的 SHA-256 是内容身份，不是模式版本，不能替代其中任何一个版本号。

## 决策

### 配置模式

1. 顶层 `version` 必填；未知版本和未知核心字段一律报错。
2. 只有“旧配置按新程序加载后语义不变”的新增可留在同一版本，例如带有保持旧行为默认值的可选字段。
3. 字段删除、改名、类型变化、默认行为变化、枚举含义变化或路径解析规则变化必须提升模式版本。
4. 新版本通过独立、可测试的显式迁移器转换；运行时不得根据字段组合猜测旧版本。
5. YAML、TOML、JSON 是同一模式的编码，不各自拥有版本；等价输入必须得到相同 `ProjectSpec`。

### 本地 IPC

1. `PROTOCOL_VERSION` 当前表示协议主版本，不假装已有尚未实现的次版本协商。
2. 新增请求、响应或事件枚举变体，删除或重解释字段，改变控制语义时提升主版本。本 ADR 对应的候选预览/应用协议把版本从 3 提升到 4。
3. 只有带 `serde(default)` 且旧端忽略后仍安全的展示字段，才允许保持主版本；控制字段不使用这种例外。
4. 客户端必须先握手再进行业务会话。版本不一致时拒绝控制；当前 CLI 会正常关闭同一安装目录中的旧 Center，安装当前可执行文件并重新握手。
5. 事件序列只在单个 Center 实例内有效；实例 ID 变化、游标越界或缓冲过期都要求重新获取快照。

### SQLite 持久化

1. `PRAGMA user_version` 是数据库模式的唯一权威版本；高于程序支持版本时只读失败，不尝试降级或覆盖。
2. 每次升级只实现 `N → N+1`，全部 DDL 与数据转换放在一个事务中，再更新 `user_version`。多级升级顺序执行相邻迁移。
3. 迁移失败必须回滚事务并保留原数据库。破坏性字段清理至少延后一个发布周期，先采用新增列/表与回填。
4. SQLite 只保存可恢复注册信息，不保存运行中 PID、run ID、日志正文或候选配置；恢复后创建新的进程身份。
5. 不支持由新版本数据库直接降级运行。需要降级时，用户应先导出受支持的配置并在隔离的 `PROCORA_HOME` 中重新注册。

## 验证要求

- 配置测试覆盖未知版本、未知字段、三格式等价和默认值保持。
- IPC 测试覆盖不兼容握手拒绝、当前版本往返和事件游标重同步；每次增加线协议变体都必须显式评审是否提升 `PROTOCOL_VERSION`。
- 存储测试覆盖新建模式、逐级迁移、事务回滚和拒绝更高版本；`STORAGE_SCHEMA_VERSION` 与最终 `user_version` 必须相同。
- 发布说明列出所有模式或协议版本变化，不能只写内部重构。

## 被否决方案

- 使用 Procora 包版本作为全部格式版本：补丁发布也会造成无意义的不兼容，且无法表达独立演进。
- 对未知字段宽松忽略：拼写错误和旧客户端控制字段可能静默改变执行语义。
- 启动时自动降级数据库：旧程序无法证明新字段不重要，存在数据损坏风险。
- 把内容 SHA-256 当配置模式版本：相同模式的每次保存都会变化，却不能说明解释规则。

## 后果

兼容性变化需要显式常量、迁移与测试，发布工作略有增加；换来的是旧修订、旧客户端和旧数据库都以可诊断方式失败，不会在后台猜测并执行。未来若确有长期独立客户端生态，可在新的协议主版本中引入 `major.minor` 与能力位；在此之前文档必须忠实描述单一主版本。
