# 中心服务器、服务宿主与任务生命周期

## 1. 运行时层级

运行时分为三个不可混淆的层级：当前用户只有一个 `Center`，它通过本地 IPC 接受 CLI/TUI 请求并持有多个 `ServiceHost`；每个 `ServiceHost` 只对应一个服务目录和一份已编译配置；每个宿主中的 `Engine` 管理该服务自己的 Task 图。

Center 使用 SQLite 持久化服务名称、目录、显式配置文件、运行期望、当前状态、错误信息、任务数和状态变更历史。中心重启时逐项重新编译配置；单个服务损坏只会使该服务进入 `failed`，不能阻止其他服务恢复。

Center 运行时通过状态数据库旁的独占锁文件保证单实例，并以独立于客户端的短周期 tick 推进 Task 退出、依赖调度和重启退避。客户端以协议版本握手获取本次进程的随机实例 ID 和事件游标；服务状态事件进入容量为 256 的内存环形缓冲。正常关闭会先封闭新控制请求、等待在途连接并停止全部宿主，SQLite 和服务目录日志均保留。中心异常退出后锁由操作系统释放，新实例通过新的实例 ID 和游标过期信号要求客户端重同步。

## 2. 单服务运行时组成

运行时由 `ServiceHost` 负责装配，核心部件包括：

- 单写者 `Engine`：处理命令、事件和状态转换。
- `ProcessRuntime`：创建、停止并观察进程树。
- `HealthRunner`：执行健康与就绪检查。
- `ResourceMonitor`：发布资源快照。
- `LogStore`：保存并发布日志帧。
- `StateRepository`：异步保存可恢复元数据。
- `EventHub`：向 IPC 会话分发快照之后的增量事件。

除引擎外的部件可以并发工作，但不得直接修改任务状态。它们只能发送带身份的结果事件。

资源采样以受管顶层 PID 为根，根据父进程关系聚合当前可识别的整棵进程树；根进程不存在时返回“不可用”，不伪造为零。`sysinfo` 以单核 100% 返回的进程树 CPU 总和会除以当前进程可用的逻辑 CPU 数，并限制为整机可用容量的 0–100%；采样结果已经映射到协议和 TUI 中可选的 CPU 与内存字段。

## 3. 身份模型

所有运行时消息至少包含以下身份中的适用部分：

- `service_name`：中心注册表中的服务稳定名称，来自配置 `project`。
- `service_root`：服务规范化目录，用于路径定位，不代替稳定名称。
- `task_id`：配置内任务稳定标识。
- `generation`：会影响进程身份的配置代次。
- `run_id`：每次进程启动生成的唯一标识。
- `command_id`：用户或自动动作的幂等标识。
- `event_seq`：daemon 实例内单调递增的事件序号。

引擎只接受与当前 generation/run 相符的异步结果。旧进程的迟到退出事件、旧健康检查结果或重连客户端重复提交的命令不能污染新状态。

## 4. 状态模型

### 4.1 服务宿主状态

服务宿主使用 `running`、`stopped`、`failed` 三个稳定状态。它表达配置是否已加载以及宿主是否具有运行期望，不汇总伪造 Task 进程状态。

### 4.2 Task 期望状态

首版保持两个稳定期望值：

- `running`：引擎应在依赖满足后维持任务运行或成功完成。
- `stopped`：引擎应确保任务进程树退出。

任务是否启用属于配置规范，不与临时用户停止混为一个布尔值。配置禁用时会生成 `stopped` 期望并阻止自动重启。

### 4.3 Task 观测状态

建议状态如下：

- `pending`：已加载但尚未具备启动条件。
- `blocked`：依赖未满足或存在明确阻断原因。
- `starting`：启动副作用已发出，等待进程身份确认。
- `running`：进程已确认存活。
- `stopping`：已请求优雅停止，等待退出。
- `exited`：进程正常结束，是否保持该状态由任务与重启策略决定。
- `failed`：启动、检查或进程退出达到失败条件。
- `backoff`：等待下一次允许重启的时间。
- `orphaned`：恢复时发现可能相关但无法确认所有权的进程。

健康状态与进程状态正交表示为 `unknown`、`starting`、`healthy`、`unhealthy`、`not_configured`，避免把每种健康组合扩张为新的进程状态。

## 5. 事件循环

```text
接收高优先级控制事件
  → 校验身份与前置条件
  → 纯状态转换
  → 生成零个或多个副作用意图
  → 更新内存快照和 event_seq
  → 异步提交副作用
  → 发布状态事件
```

控制事件包括用户命令、配置提交、进程启动/退出和定时器到期。日志帧和高频指标不进入核心状态队列，仅在需要触发健康或策略判断时生成低频摘要事件。

纯状态转换函数应可用表驱动测试覆盖。所有系统调用在状态转换之外执行；失败以新事件返回，不能在持有状态写锁时等待 I/O。

## 6. 启动调度

启动一次任务的基本流程：

1. 对账发现 `desired=running` 且状态允许启动。
2. 检查全部依赖边条件；不满足则记录可解释的阻断集合。
3. 预留新的 `run_id`，状态转换为 `starting`。
4. 向进程端口提交包含完整规范快照的启动意图。
5. 收到已创建事件后转换为 `running`，启动日志、监测和检查会话。
6. 依赖 `started` 的下游可继续调度；依赖 `healthy` 的任务等待健康事件。

拓扑序用于得到确定性调度结果，但没有依赖关系的任务可按配置的并发上限平行启动。并发限制需要同时支持项目级上限和启动风暴保护。

## 7. 停止调度

停止项目时按照反向依赖顺序进行，确保下游先于其依赖退出：

1. 将目标任务期望状态设为 `stopped`，取消待启动与重启定时器。
2. 停止新的健康检查和非必要采样，但继续读取输出。
3. 发送平台对应的优雅停止操作。
4. 等待任务配置的宽限期。
5. 超时后终止整个受管进程树。
6. 排空剩余 stdout/stderr，刷新日志索引并记录退出结果。

文件日志索引与正文一起保存在所属 Service 的 `.procora/logs`。游标由“活动文件代次 + 字节偏移”组成；轮转会推进代次，旧游标再次读取时返回 Gap 并从当前活动文件的有界尾部恢复。gzip 归档同时受数量、总压缩字节和最长保留时间约束，Center 数据目录不保存这些文件。

单个任务停止失败不能永久阻塞整个项目关闭。引擎记录失败并继续处理已达到超时的其他任务，最终汇总非零结果。

## 8. 重启与退避

重启策略至少包含：

- `never`：退出后不重启。
- `on-failure`：启动失败、非零退出或策略定义的异常退出时重启。
- `always`：只要期望仍为 `running` 就重启。

当前退避采用以 `restart_delay_ms` 为初值、30 秒封顶的指数策略。`max_restarts` 默认 0 表示无限，正数限制当前 generation 和连续退出窗口内实际允许的自动重启次数；初始启动不计入。达到上限后不再创建进程，TUI 会显示最终退出结果和耗尽次数。手动 `start` 会开启新 generation 并清零计数；原地放宽上限则从当前计数继续恢复调度。

重启计数绑定 generation；影响进程身份的配置更新会开启新 generation，避免旧配置失败次数阻止新配置启动。`ServiceHost` 使用单调时钟记录每个真实进程的运行时间；只有达到 `restart_reset_after_ms`（默认 60 秒）的运行才清零连续计数，创建失败按 0 毫秒处理。设置 0 可禁用稳定窗口重置。手动停止、配置禁用和 daemon 正常关闭永远不触发自动重启。少量抖动仍是后续用于多服务同步故障的计划能力。

## 9. 健康检查

引擎和 `ServiceHost` 已支持带 `generation/run_id` 的 exec/HTTP GET readiness 检查与 `healthy` 依赖调度。没有检查器的 Task 使用 `not_configured`，其 `healthy` 依赖退化为“受监管进程已经创建”。每次检查具有独立超时，同一 Task 同时最多存在一个检查，一个 Service 同时最多存在 32 个活动或已取消但尚未返回的检查；结果槽容量固定为一。Task 停止或退出时取消检查，exec 会回收进程树，HTTP 请求不会阻塞停止路径且仍受原总超时约束；旧 run 的结果不能作用于新 run。健康状态由连续成功/失败阈值决定：

- 启动宽限期内失败不立即标记 unhealthy。
- 达到成功阈值后发布 healthy。
- 达到失败阈值后发布 unhealthy。
- 超时视为失败并主动回收检查进程。

HTTP GET 支持 HTTP/HTTPS、可选端口、路径、请求头和精确预期状态码，不跟随重定向。当前 unhealthy 只影响尚未启动的依赖并显示告警，不默认重启主任务。startup 检查和“健康失败触发重启”仍是计划能力；重启必须作为显式策略，不能让暂时网络故障制造重启风暴。

## 10. 配置对账

配置差异分为：

- `add`：注册新任务并按期望状态调度。
- `remove`：停止旧任务后从活动图移除。
- `restart`：命令、参数、环境、目录或进程级策略变化。
- `update_in_place`：日志保留、指标间隔、标签等可在线更新字段。
- `unchanged`：语义等价，不因格式或字段顺序变化重启。

提交新图之前先完成全量验证。对账时必须同时考虑旧图和新图的依赖关系：停止按旧图反向顺序，启动按新图拓扑顺序。若部分启动失败，已生效配置修订仍保持可追踪；引擎通过正常重启/失败状态继续对账，不伪造事务性进程回滚。

## 11. TUI 与后台模式

`procora` 是全局中心的服务总览 TUI 入口，会在需要时启动中心，并可筛选、按名称/状态/CPU/内存排序、查看 Service 聚合资源、管理全部注册服务或进入单服务 Task 详情后返回。`procora PATH` 继续直接打开指定路径：已有全局服务器时读取对应服务快照，目标尚未注册时先注册；全局服务器不存在时由用户选择启动全局服务或创建与 TUI 同生命周期的临时 `ServiceHost`。

`procora add <path>` 是持久托管入口。该命令必要时启动当前用户唯一的中心服务器，通过本地 IPC 注册服务；终端命令退出不会停止中心进程。`procora show <name/path>` 只连接已经注册的服务，不隐式注册新路径。

当前 TUI 已区分临时服务和全局服务。全局模式通过协议提交控制命令、续读 Service 本地 Task 日志、拉取增量事件和资源快照；临时模式通过进程内实时会话提供相同能力。

实时循环把输入等待上限设为 50ms，服务快照默认每 500ms 检查，日志只在日志页按 50ms 间隔续读；单批最多读取 1 MiB，单轮最多连续追赶 16 批并合并后更新视图。只有输入、Resize、快照、日志或反馈实际变化时才重绘。低能力终端可通过 `PROCORA_TUI_PLAIN=1` 强制使用 ASCII 边框、符号和无彩色输出，`NO_COLOR` 与 `TERM=dumb` 也会自动启用该模式。

## 12. 中心服务器恢复与关闭

中心服务器启动时读取服务注册表，并逐项重新编译记录的显式配置文件。配置有效的服务恢复原运行期望；配置缺失或无效的服务保留注册信息并进入 `failed`，其他服务继续恢复。

当前 SQLite 数据库保存服务名称、目录、配置文件、运行期望、当前状态、错误信息、任务数和状态历史。日志正文不进入数据库，而是留在对应 Service 的 `.procora/logs` 目录。当前不持久化或接管 Task PID；Center 恢复时重新编译配置，并为运行期望中的 Task 创建全新的 `generation/run_id`。旧 run 的迟到事件会被身份校验拒绝。未来若增加跨进程接管，必须同时验证配置修订、平台启动身份和运行令牌，不能仅凭 PID 接管或终止。

正常关闭协议已经实现：`procora down` 请求 Center 停止，Center 按服务逐项触发 Task 反向停止、排空输出并释放 IPC 端点，同时保留 SQLite 中的运行期望，供下次 `up` 恢复。嵌入式 TUI 的 `q`、Esc 与 Ctrl-C 走同一宿主停止路径。
