# 测试与质量约束

## 1. 基本规则

项目遵循以下仓库约束：

- 注释、文档和测试场景说明使用中文。
- 单个代码文件不超过 500 行，接近上限时按职责拆分。
- 每个关键 trait、结构体、函数和静态全局变量前提供简短注释。
- 测试统一放在 `tests/` 目录，不在实现文件尾部堆积大型测试模块。
- 每个关键状态转换、错误恢复和平台边界都有自动化测试。

所有公共行为、二进制和端到端测试都位于根 `tests/`。共享构造器和夹具位于 `tests/support/` 与 `tests/fixtures/`；测试文件按其覆盖的模块命名，避免把测试逻辑复制到多个位置。

## 2. 测试层次

### 2.1 领域测试

针对纯逻辑快速执行：

- 任务图环检测、拓扑序和反向停止顺序。
- 状态机的合法/非法转换。
- 依赖条件传播。
- 重启退避、计数重置与时间边界。
- 配置语义差异分类。

优先使用表驱动和性质测试。时间、ID 与随机抖动通过注入的 clock/ID/random 端口控制，禁止依赖真实 sleep。

`tests/properties.rs` 使用固定种子生成 192 个、最多 28 个 Task 的随机 DAG，验证每次创建都已满足直接依赖、停止顺序始终是反向依赖序，并验证配置自差异不产生影响集合。固定种子保证失败可重放，不把随机偶发现象留给 CI。

### 2.2 配置契约测试

- 同一语义的 YAML、TOML 和 JSON 产生相同 `ProjectSpec`。
- 默认值、profile、include、环境和 CLI 覆盖符合固定优先级。
- 错误包含来源、字段路径和可读建议。
- 路径按声明文件目录解析。
- 未知字段、循环依赖和缺失依赖被拒绝。
- Python 辅助进程测试隔离环境、超时整树回收、非零退出、脚本/stdout 上限、严格单 JSON 与生成输出修订。

规范配置样例放入 `tests/fixtures/config/`，并明确区分 valid、invalid 与 equivalent。

### 2.3 适配器契约测试

每个端口提供可复用契约套件，mock 与真实适配器接受相同核心断言：

- `ProcessRuntime`：启动身份、退出事件、优雅停止、强制回收与多次停止幂等。
- `LogStore`：Service 本地路径、Service/Task 隔离、gzip 滚动、游标、保留和中断恢复。
- `ResourceMonitor`：单位、树聚合、不可用能力和进程退出竞态。
- `StateRepository`：SQLite 事务、状态历史去重、模式版本拒绝和损坏数据库处理。
- `DefinitionSource`：版本稳定、重复事件去抖和内容哈希。

include 契约额外覆盖三格式片段优先级、跨文件依赖、各声明文件的路径基准、循环/深度/数量/总字节限制、符号链接越界、缺失成员创建恢复，以及 preview/apply 对整个闭包的 TOCTOU 校验。

Git 来源契约使用真实本地仓库覆盖浅获取、完整 commit 固定、跨格式 include、可变引用修订变化、重新确认、无效候选保留、缓存篡改、归档上限，以及协议/引用/路径注入拒绝。网络测试不依赖公共服务；远端认证和故障注入需要后续使用本地受控 Git/SSH 服务夹具。

### 2.4 集成与端到端测试

使用小型测试子进程覆盖：

- 按节奏输出 stdout/stderr。
- 忽略优雅停止并派生后代。
- 在信号后正常退出。
- 消耗受控 CPU/内存。
- 健康检查由失败转成功或超时。

端到端场景从配置输入开始，经过 daemon、IPC 到 CLI/TUI 视图模型，验证启动依赖、日志续读、客户端重连、配置热更新和反向停止。

## 3. 关键状态矩阵

至少覆盖以下容易发生竞态的节点：

| 场景 | 必须验证 |
| --- | --- |
| 启动期间收到停止 | 不进入 running，已创建进程最终被回收 |
| 旧 run 退出事件迟到 | 新 run 状态不受影响 |
| 配置解析失败 | 旧修订继续运行，不产生部分更新 |
| 依赖健康后又退化 | 未启动下游阻塞，已运行下游按策略处理 |
| 日志客户端过慢 | 任务输出不阻塞，客户端收到 Gap 并能补读 |
| daemon 异常重启 | 不凭裸 PID 错误接管其他进程 |
| 停止宽限期超时 | 整个受管进程树被强制回收 |
| 监测接口缺失 | 返回 unavailable，UI 不显示伪造零值 |

## 4. 三平台 CI 矩阵

基础矩阵应覆盖 stable Rust 的 Linux、macOS 和 Windows：

```text
格式与静态检查
  → 领域/配置快速测试
  → 三平台适配器测试
  → 三平台端到端测试
  → 发布构建与最小启动检查
```

Release workflow 在 Linux、macOS 和 Windows 原生 runner 上执行全部测试；Linux 分别覆盖默认 feature 与 `systemd` feature，其余发布架构至少完成构建验证。平台特有用例通过 `#[cfg]` 在对应 runner 上运行：macOS 覆盖 LaunchAgent 渲染，Windows 覆盖任务计划程序、Job Object 与 UTF-16 路径。仅因断言不稳定而跳过测试不可接受；应改用事件同步、虚拟时钟或范围断言消除偶发性。

## 5. 质量门禁

单一 crate 建立后，提交前的目标命令为：

```bash
cargo fmt --all -- --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test --all-features
cargo doc --no-deps
cargo deny check bans licenses sources
cargo deny check advisories
```

本地以这些命令作为质量门禁；`ci.yml` 固定 Rust 1.95 最低版本，并在 Ubuntu 单独执行 systemd feature、显式总线和权限失败回滚测试；`security.yml` 在依赖变化和每周计划中检查许可证、来源和 RustSec 漏洞，六目标 Release workflow 负责补充三平台构建验证。涉及不安全代码时，必须在局部模块说明安全不变量并增加专门测试；默认不允许为了绕过平台库限制扩大 `unsafe` 范围。

## 6. 性能与稳定性测试

在功能稳定后建立以下基准与压力场景：

- 100 个小任务同时启动时的调度延迟和事件队列峰值。
- 单任务高速 stdout/stderr 下的吞吐、内存上限和丢失策略。
- 多个观察客户端同时订阅日志与指标时的 daemon 开销。
- 大任务图的编译、差异计算和 TUI 刷新时间。
- 24 小时运行中的句柄、文件描述符、任务和内存泄漏。

当前快速压力门禁由 `tests/log_stress.rs` 覆盖：8 个并发写者写入 2000 条唯一记录并检查无丢失/撕裂，同时连续追加 256 KiB 触发密集 gzip 轮转，验证归档上限和临时文件清理。它不替代 24 小时 soak；长期运行仍需在定时 CI 或发布候选环境记录进程资源曲线。

`soak.yml` 每周以 release 模式运行默认测试中忽略的 `tests/soak.rs`，连续完成 2000 次真实 Task 启动、运行状态观察和停止，验证每轮状态闭环，并在 Unix 检查文件描述符没有持续增长。循环数可用 `PROCORA_SOAK_CYCLES` 调整但限制在 1–10000；它用于尽早发现线性泄漏，仍不替代发布候选的 24 小时内存、线程和句柄曲线。

基准结果用于确定默认队列容量、采样间隔和日志批量大小，不把特定开发机数字写成跨平台硬性承诺。
