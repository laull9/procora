# 安全策略

## 支持范围

Procora 仍处于早期版本。安全修复优先进入最新发布版本；旧版本不承诺长期维护。Release 页面和变更日志会明确标出安全相关修复。

## 报告方式

请优先使用 GitHub 仓库的私密漏洞报告功能，不要在公开 Issue、讨论区或日志附件中披露可利用细节、令牌、用户名、服务路径或业务配置。

报告应尽量包含：

- 受影响版本、操作系统和 CPU 架构。
- 最小复现配置和操作步骤。
- 预期边界与实际影响。
- 是否需要本机同用户权限、恶意配置或网络位置。
- 已知缓解方法；不要附带真实凭据或敏感业务数据。

维护者确认前，请避免公开利用代码。收到报告后应先确认影响范围和可复现性，再协调修复、回归测试、公告与披露时间。

## 默认安全边界

- Center 默认只通过当前用户的本地 IPC 接收请求，不监听 TCP 端口。
- Unix 校验对端用户；Windows Named Pipe 使用当前用户 DACL。
- IPC 限制帧大小、连接并发和读写期限。
- Task 与健康检查默认使用程序加参数数组，不经过隐式 shell。
- 下载依赖支持 SHA-256；远程来源应固定校验和。
- 归档解包拒绝绝对路径、父目录和安装根目录外条目。
- Center 重启不凭裸 PID 接管或终止进程。
- `procora.py` 只在用户显式指定精确文件路径时执行；目录发现不会自动执行代码。

这些边界不等于对 Task 配置的沙箱。运行 Procora 配置等同于授权其中的程序以当前用户权限执行；不要运行来源不可信的配置或二进制。

Python 配置同样是可信代码执行。辅助进程会清空继承环境、关闭 stdin，限制脚本和输出大小、设置 5 秒超时，并由进程组或 Job Object 负责整树回收；这些措施隔离崩溃、挂起和输出洪泛，但不能阻止脚本按当前用户权限访问文件、网络或其他系统资源。审查脚本后再显式运行，且不要把该机制当作沙箱。

Git 定义源默认只生成候选，不注册服务或启动 Task。远端只允许无内嵌凭据的 HTTPS/SSH/SCP，禁用交互、全局配置、凭据 helper、hooks、外部 remote helper 和远端 file 协议；对象库、命令时间、输出、文件数、checkout 与缓存都有上限。引用最终固定为完整 commit，确认阶段必须重新获取并核对组合修订。Git commit 哈希提供内容身份，不等于作者可信、签名验证或仓库授权；当前也不提供私有仓库凭据代理。

## 供应链门禁

`deny.toml` 只允许 crates.io 来源并维护许可证策略。CI 在依赖变化和每周计划任务中运行 cargo-deny 与 RustSec；最低 Rust 版本和完整质量检查由独立 CI 验证。任何漏洞忽略、许可证例外或 Git 依赖都必须附带原因、限定版本和移除条件。
